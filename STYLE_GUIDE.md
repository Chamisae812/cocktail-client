# Project Style Guide

이 문서는 이 프로젝트에서 일관된 방식으로 코드를 작성하기 위한 기준이다.
완벽한 규칙집보다는, 실제 코드 작성할 때 바로 참고할 수 있는 실용적인 가이드를 목표로 한다.

## 1. 기본 방향

- 코드는 배우기 쉽고 읽기 쉽게 작성한다.
- 과한 추상화보다, 한 번에 이해되는 명확한 구조를 우선한다.
- React의 기본 패턴을 충실히 사용한다.
- 작은 프로젝트에서는 구조를 단순하게 유지한다.
- 코드를 처음 보는 사람도 흐름을 따라갈 수 있게 작성한다.

## 2. 파일과 폴더 구조

- 페이지 단위 컴포넌트는 `src/pages`에 둔다.
- 공통 UI 컴포넌트는 `src/components`에 둔다.
- 라우터 설정은 별도 파일로 분리한다.
- 전역 스타일은 공통 CSS 파일에서 관리한다.
- 프로젝트가 작을 때는 폴더를 필요 이상으로 잘게 나누지 않는다.

## 3. 컴포넌트 작성 방식

- 함수형 컴포넌트를 사용한다.
- 컴포넌트 이름은 역할이 보이도록 짓는다.
- 한 파일에는 한 컴포넌트의 책임이 잘 드러나게 작성한다.
- 로직이 아주 복잡하지 않다면, 컴포넌트 안에서 바로 처리해도 된다.
- 억지로 커스텀 훅이나 유틸 함수로 분리하지 않는다.

예시:

- `MainPage`
- `LoginPage`
- `RecipeAddPage`
- `Layout`
- `Header`

## 4. 네이밍 규칙

- 변수, 함수, state 이름은 역할이 바로 보이게 짓는다.
- 축약형보다 의미가 분명한 이름을 선호한다.
- 이벤트 핸들러는 `handle`로 시작한다.
- 데이터를 가져오는 함수는 `get` 또는 동작이 드러나는 이름을 사용한다.
- state와 setter는 React 기본 규칙을 따른다.

예시:

- `getRecipes`
- `handleSubmit`
- `recipes`, `setRecipes`
- `description`, `setDescription`

## 5. 상태 관리

- 간단한 화면 상태는 `useState`로 관리한다.
- 화면 진입 후 필요한 데이터 호출은 `useEffect`를 사용한다.
- 상태는 필요한 만큼만 두고, 불필요하게 쪼개지 않는다.
- 아직 규모가 작다면 전역 상태 관리 도구를 성급히 도입하지 않는다.

## 6. 비동기 처리

- `fetch`와 `async/await`를 사용해 직관적으로 작성한다.
- 데이터 요청 흐름은 한눈에 읽히게 유지한다.
- 처음에는 단순하게 작성하되, 실제 기능이 늘어나면 에러 처리와 로딩 처리도 함께 보강한다.
- 화면 이동이 요청 완료에 의존한다면 `await`로 완료 시점을 맞춘다.

권장 예시:

```js
const getRecipes = async () => {
  const response = await fetch("http://localhost:4000/recipes");
  const data = await response.json();
  setRecipes(data);
};
```

## 7. 주석 스타일

- 주석은 "이 코드가 무엇을 하는지"보다 "왜 이렇게 쓰는지"가 필요할 때만 적는다.
- 학습 중인 개념은 짧게 설명해도 된다.
- 주석이 너무 길어져서 코드보다 더 눈에 띄지 않게 주의한다.
- 코드만으로 충분히 읽히는 부분은 굳이 주석을 달지 않는다.

좋은 방향:

- React 문법이나 훅의 의미를 짧게 설명
- 처음 보는 사람이 헷갈릴 수 있는 포인트 설명

피하고 싶은 방향:

- 코드 한 줄을 그대로 한국어로 다시 설명하는 주석
- 오래되면 쉽게 틀릴 수 있는 설명

## 8. 라우팅

- 라우팅 구조는 별도 파일에서 한 번에 보이게 관리한다.
- 페이지 컴포넌트는 라우터에서 명확하게 연결한다.
- 공통 레이아웃이 있으면 `Layout`과 `Outlet` 패턴을 사용한다.

## 9. 스타일링

- 공통 스타일은 전역 CSS에 둔다.
- 컴포넌트 전용 스타일은 가까운 CSS 파일로 분리할 수 있다.
- class 이름은 역할이 보이게 작성한다.
- 너무 짧거나 의미 없는 class 이름은 피한다.

예시:

- `page-shell`
- `page-content`
- `site-header`
- `site-header__nav`

## 10. 문체와 표현

- UI 문구와 주석은 한국어를 사용해도 된다.
- 다만 파일 인코딩은 항상 UTF-8로 맞춘다.
- 한글이 깨지지 않도록 에디터와 저장 인코딩을 확인한다.

## 11. 일관성 규칙

- 새 코드는 기존 파일의 스타일을 최대한 따른다.
- 한 파일 안에서는 따옴표, 들여쓰기, 세미콜론 스타일을 최대한 일관되게 유지한다.
- 프로젝트 전체에서 완벽한 통일보다는, 수정하는 파일 내부의 일관성을 우선한다.

## 12. 지금 프로젝트 기준으로 Codex가 따라야 할 원칙

- 과하게 어렵게 만들지 않는다.
- 처음 보는 사람도 읽기 쉽게 작성한다.
- 필요한 기능을 먼저 완성하고, 추상화는 나중에 고민한다.
- React 기본 문법과 흐름을 중심으로 구현한다.
- 함수명과 state 이름은 설명적으로 작성한다.
- 사용자의 기존 코드 톤을 해치지 않는 범위에서만 개선한다.
- 실전에서 꼭 필요한 부분은 자연스럽게 보완한다.
  - 예: `await` 누락 보완
  - 예: 기본적인 에러 처리 추가
  - 예: 지나치게 깨진 인코딩 수정

## 13. 이 문서의 사용 방법

- 새 기능을 만들기 전에 이 문서를 먼저 확인한다.
- 기존 코드와 이 문서가 충돌하면, 우선 현재 코드베이스의 실제 패턴을 따른다.
- 프로젝트가 커지면 이 문서를 조금씩 업데이트한다.

이 문서는 고정된 규칙집이 아니라, 프로젝트와 함께 다듬어 가는 기준 문서다.
